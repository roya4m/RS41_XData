Scripting Architecture

ScriptExecutor provides interface between IronPython scripts and the sounding system. See following illustation:

Scripting Architecture
  • ScriptExecutor.exe
    • Runs an IronPython script instance. The main script may include additional files.
    • Provides ISounding methods to the script
    • Requests notifications from DataService and forwards those to the script
    • Uses MessageService methods for message handling
    • Reads parameter values for the script
    • Writes log file to same folder where scripts is running. Note! Log file is named as [script name]*.log, so do not write same file from the script.
    • Provides for each ScriptExecutor.exe instance a queue of events in which the order is guaranteed. This queue will fill up (exception in [script name]*.log after overflow) in case the script performs heavy computations or I/O operations after receiving an event.
    • Provides configuration flags in ScriptExecutor.exe.config:
      • DebugPython (default False): If True, debug mode will be passed to python ScriptRuntime, so that python scripts can be debugged.
      • SendAllPythonEvents (default False): If True, exceptions occurring in IExecutableScript.HandleData will be reported as an event to GUI
  • MainCustomer.py
    • Provides IExecutableScript interface for the sounding system calls
      • Get properties for metadata: script name, description, version, etc.
      • HandleData method for handling requested notifications
    • Can use ISounding methods to
      • read parameter values
      • read information from sounding database
      • send messages
      • store ozone calculation results to sounding database
    • Sounding system and IronPython dependent
  • IncludeA.py
    • Optional script include files which will be expected in the same folder as the main script (MainCustomer.py)
    • Re-usable data processing and algorithms
    • Preferably no dependencies to Sounding system
  • ControlService
    • Starts and stops ScriptExecutor based on configuration. Scripts are started when sounding is created. Cleaning of running scripts are done before starting script again.
  • GUI
    • Provides methods for user to upload new scripts and possibility choose what scripts should be run.
    • The GUI can receive custom PythonScriptInfoEvents. Be aware, that each event is persisted to the database. Therefore, the event should not be send from (long running) loops.
